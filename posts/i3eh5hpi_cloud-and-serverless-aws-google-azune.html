[Cloud and Serverless] AWS, Google, Azune

By Steven,

In Amazon Web Service Side

The most commonly used AWS services and components in cloud solutions as of 2025 include foundational compute, storage, database, networking, and serverless services that are integral to building scalable and flexible cloud architectures.

Most Used AWS Services

  • Amazon EC2 (Elastic Compute Cloud): Provides scalable virtual servers for running applications in the cloud.
  • AWS Lambda: Serverless compute service that runs code in response to events without managing servers.
  • Amazon S3 (Simple Storage Service): Highly scalable object storage for any kind of data.
  • Amazon RDS (Relational Database Service): Managed relational databases, supporting engines like MySQL and PostgreSQL.
  • Amazon DynamoDB: Fully managed NoSQL database with low latency and high scalability.
  • Amazon ECS and EKS: Container orchestration services for Docker containers and Kubernetes.
  • Amazon API Gateway: Fully managed service for creating, deploying, and managing APIs.
  • Amazon CloudFront: Content delivery network for fast content delivery globally.
  • Amazon Elastic Beanstalk: Platform service for easy application deployment and management.
  • Amazon Aurora: High-performance, MySQL and PostgreSQL-compatible relational database.
  • Amazon ElastiCache: Managed in-memory caching service to improve data retrieval speeds.
  • IAM (Identity and Access Management): Security service for managing user permissions and access.
  • Amazon VPC (Virtual Private Cloud): Enables provision of isolated cloud resources within the AWS network.

These services are widely used because they cover essential computing, storage, database, and security needs and support both traditional and serverless architectures. They also offer frameworks for containerized applications, microservices, and API management, making them highly versatile in cloud solution designs.

In Google Cloud Service

The mostly used services and components in Google Cloud Platform (GCP) solutions in 2025 focus largely on computing, data management, machine learning, and networking, reflecting Google's strong emphasis on AI and data analytics.

Most Used Google Cloud Services

  • Google Compute Engine: Provides scalable virtual machines for diverse workloads.
  • Google Kubernetes Engine (GKE): Managed Kubernetes service for container orchestration.
  • Google Cloud Storage: Durable and highly available object storage service.
  • BigQuery: Fully managed data warehouse for large-scale data analytics.
  • Cloud Functions: Serverless compute service to run event-driven code.
  • Cloud Pub/Sub: Messaging service for event ingestion and delivery.
  • Cloud SQL: Managed relational database service supporting MySQL, PostgreSQL, and SQL Server.
  • Vertex AI: Integrated machine learning platform for building, deploying, and scaling AI models.
  • Identity and Access Management (IAM): Access control and security management.
  • Cloud Networking (VPC, Cloud Load Balancing): Provides network segmentation, routing, and load balancing.
  • Operations Suite (formerly Stackdriver): Monitoring, logging, and diagnostics for cloud applications.

Google Cloud is particularly known for its innovations in AI with services like Vertex AI and deep integrations with Google Workspace and open-source tools. Its services are widely used in data-centric and AI-driven applications.

The mostly used Microsoft Azure services and components in cloud solutions for 2025 focus on scalable computing, container orchestration, AI, storage, and DevOps, reflecting Azure’s strong enterprise cloud presence.

In Microsoft Azure Service

Top Azure Services in 2025

  • Azure Virtual Machines (VMs): Scalable cloud-based Windows and Linux servers, core to many cloud deployments.
  • Azure Kubernetes Service (AKS): Managed Kubernetes for containerized app deployment and orchestration.
  • Azure Blob Storage: Highly scalable object storage for unstructured data like backups, logs, and media.
  • Azure AI and Machine Learning: Pre-built AI models and ML services integrated deeply with Azure’s cloud tools.
  • Azure Functions: Serverless compute for event-driven code execution without managing infrastructure.
  • Azure SQL Database: Managed relational database with performance intelligence and scaling features.
  • Azure Cosmos DB: Globally distributed multi-model database with low latency and high availability.
  • Azure DevOps: Full-stack DevOps platform with CI/CD pipelines, agile planning, and version control.
  • Azure Logic Apps: Code-free workflow automation connecting many enterprise services.
  • Azure Monitor and Application Insights: Monitoring and diagnostics platform for maintaining app performance.

These services cover essential infrastructure, app hosting, AI, database, and development lifecycle capabilities that enterprises rely on. Azure also puts strong emphasis on AI, security, and hybrid cloud scenarios, making these services widely adopted in 2025.

AWS Lambda / Serverless

UAC / OAuth2

To implement User Access Control (UAC) with both traditional Username-Password and OAuth2 in a web app using AWS Lambda, a common and effective approach is to use Amazon Cognito combined with AWS Lambda.

Key Components and Approach

  • Amazon Cognito User Pools provide built-in user directory and authentication capabilities which support traditional username-password login as well as federated sign-in via OAuth2 (including social identity providers).
  • AWS Lambda Triggers in Cognito can be used to customize authentication flows, handle sign-up, sign-in, and token generation.
  • For traditional username-password authentication, users are managed in Cognito user pools with password policies, multi-factor authentication (MFA), and secure handling.
  • For OAuth2, Cognito supports authorization flows with external identity providers and can issue OAuth2 tokens (access/ID/refresh tokens). It handles the OAuth2 protocol details.
  • Your Lambda functions can be used as authorizers to validate JWT tokens issued by Cognito before granting access to your APIs or resources.
  • For custom OAuth2 use cases like device grant flows, Lambda combined with DynamoDB can implement storage and flow control logic, but Cognito covers most standard OAuth2 scenarios.

How It Works

  1. User sign-up/sign-in via Cognito user pool UI or API, supporting username-password or federated OAuth2.
  2. Cognito issues JWT tokens upon successful authentication.
  3. API requests to your backend Lambda functions carry these tokens (usually in Authorization headers).
  4. Use Lambda Authorizer functions or API Gateway authorizers to validate tokens before allowing resource access.
  5. Use Cognito triggers (Lambda functions) to add custom logic in sign-up/sign-in flows if needed.

Summary

Using Amazon Cognito combined with AWS Lambda offers a robust way to implement UAC that supports both username-password and OAuth2. Cognito manages identity, authentication, and token handling, while Lambda functions can customize flows and enforce authorization on APIs or backend services. This architecture simplifies security management in serverless web apps.

If a fully customized OAuth2 server is needed, Lambda functions with supporting data stores such as DynamoDB can be developed; however, for most cases, Cognito's managed services cover the requirements effectively.aws.amazon+4

Here is a flowchart illustrating the user access control process in a web app using AWS Lambda with traditional Username-Password and OAuth2 authentication:

  • Users choose to sign up/sign in either with a username-password pair or via OAuth2 through external identity providers.
  • Amazon Cognito User Pool handles authentication and issues JWT tokens.
  • API requests from the user include these tokens when calling backend AWS Lambda functions.
  • A Lambda authorizer validates the tokens before granting access to resources.
  • Lambda triggers can be optionally used to customize authentication logic during sign-up and sign-in.

This flow outlines how Cognito, Lambda, and OAuth2 work together to enable secure user access control in a serverless web app environment.
